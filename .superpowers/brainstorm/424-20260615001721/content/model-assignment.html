AI Model 分配 & Task 拆分

以 Next.js + Prisma + PostgreSQL 為基礎, 按任務複雜度和模型特性分配

Phase 1: Foundation (需先完成)

Prisma Schema + Seed

Model: DeepSeek V4 Pro

Data model 較複雜, 多表關聯, 需要強推理能力確保 schema 正確

Foundation | 不可並行

Phase 2: Feature Tracks (可並行)

1

Auth + Register/Login

Model: Qwen 3.7 Max

NextAuth.js v5 整合, Google OAuth, Email/Phone login, JWT session. 框架知識需求高

依賴: Schema
2

Home Page

Model: Qwen 3.7 Plus

Landing page UI, Logo, Slideshow, 4 plans display, Footer, Map. 純展示, 無複雜邏輯

獨立
3

Booking Calendar

Model: DeepSeek V4 Pro

最複雜: 週視圖, 時段分欄, 顏色編碼, 剩餘座位計算, 假期判斷, 手機響應式. 需要強邏輯推理

依賴: Schema + Auth
4

Admin Panel

Model: Kimi K2.6

消費計劃 CRUD, 支付設定 (QR upload), 會員列表, 系統設定. 較多 CRUD 表單操作

依賴: Schema + Auth
5

Account + History

Model: Qwen 3.7 Plus

會員資料顯示, 歷史預約列表, 上傳單據按鈕. 中等複雜度

依賴: Schema + Auth
6

Payment Flow

Model: MiniMax M3

付款版面, QR code 顯示, 金額計算, 上傳單據 (png/jpg). 表單驗證 + 檔案處理

依賴: Booking
7

i18n 繁簡英

Model: Mimo V2.5 Pro

所有頁面三語翻譯檔, next-intl 整合

獨立 (最後套用)

Phase 3: Verification

Global Verification

Model: DeepSeek V4 Flash

速度優先的驗證模型. 檢查所有 feature 的 API 兼容性, TypeScript 型別, 全域一致性, E2E flow 完整性

Task Dependency Flow

Schema (DSv4Pro) → Auth (Q3.7M) Home (Q3.7P) Calendar (DSv4Pro) Admin (Kimi) Account (Q3.7P) → Payment (MiniMax)

i18n (Mimo) ← 最後全頁套用

Verification (DSv4Flash)

Model 特性理由

DeepSeek V4 Pro (x2 tasks)

  • 最強推理能力 → Schema 設計 + Calendar 複雜邏輯

Qwen 3.7 Max

  • 框架整合經驗最廣 → NextAuth.js 設定

DeepSeek V4 Flash

  • 速度快、Cost 低 → 適合全域檢查/驗證

Kimi K2.6 / MiniMax M3 / Mimo V2.5 Pro

  • 一般 CRUD / UI 任務, 成本效益好
